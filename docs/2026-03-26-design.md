# gh-claudecode — Design Spec

**Date:** 2026-03-26
**Status:** Approved

## Overview

Two independent shell scripts that automate GitHub issue lifecycle using Claude Code CLI. The **Refiner** interacts with humans to clarify issues until a checklist is complete. The **Solver** picks up approved issues and implements them following Gitflow.

## Goals

- Automate issue triage and refinement through conversational AI
- Automate implementation of approved issues (worktree, branch, code, PR)
- Generic — works with any GitHub repository
- Runs locally now, Kubernetes later

## Issue Lifecycle (Labels)

Agents control issue state exclusively via labels. The GitHub Project Board is for human visualization only — agents never interact with it.

| Label | Meaning | Set by |
|-------|---------|--------|
| *(none)* | New issue, needs refinement | Human creates issue |
| `refining` | Refiner is interacting with human | Refiner |
| `ready` | Checklist complete, awaiting approval | Refiner |
| `approved` | Approved for implementation | Human |
| `in-progress` | Solver is working on it | Solver |
| `done` | PR created | Solver |
| `failed` | Solver timed out or failed | Solver |

## Components

### 1. Refiner (`refiner.sh`)

Monitors GitHub issues and interacts with humans until a mandatory checklist is fully completed.

**Parameters:**
- `--repo` (required) — GitHub repo in `owner/repo` format
- `--interval` (optional, default: 300) — polling interval in seconds
- `--model` (optional, default: claude-sonnet-4-6) — Claude model to use

**Behavior:**

| Issue state | Action |
|-------------|--------|
| No labels | Add label `refining`, comment with initial questions to fill checklist |
| Label `refining`, last comment by human | Analyze response with `claude` CLI, continue filling checklist, comment with follow-up questions |
| Label `refining`, last comment by bot | Skip — wait for human response |
| Checklist complete | Update issue body with filled checklist, replace label `refining` → `ready` |

**Mandatory Checklist:**

Every issue must have all items filled before being marked `ready`:

- [ ] **Problem described** — clear description of the bug or need
- [ ] **Proposed solution** — how to solve it
- [ ] **Affected files** — which files/crates/modules are impacted
- [ ] **Acceptance criteria** — how to verify the fix/feature works
- [ ] **Type** — bug / feature / enhancement
- [ ] **Complexity estimate** — low / medium / high

### 2. Solver (`solver.sh`)

Picks up approved issues and implements them autonomously.

**Parameters:**
- `--repo` (required) — GitHub repo in `owner/repo` format
- `--interval` (optional, default: 600) — polling interval in seconds
- `--timeout` (optional, default: 3600) — max time in seconds for an issue to stay `in-progress` before marking as `failed`
- `--model` (optional, default: claude-sonnet-4-6) — Claude model to use

**Behavior:**

1. Query issues with label `approved`
2. For each issue:
   a. Replace label `approved` → `in-progress`, comment with start timestamp
   b. Clone/fetch repo, create git worktree from `develop`
   c. Create branch: `bugfix/issue-{N}-description` or `feature/issue-{N}-description` (based on type from checklist)
   d. Run `claude` CLI with issue context (title, body, checklist, comments) to implement the solution
   e. Commit with message referencing issue, push branch
   f. Create PR targeting `develop` with `Closes #N`
   g. Replace label `in-progress` → `done`
3. If implementation fails, comment on issue with error details, replace label `in-progress` → `failed`

**Stale detection:**

On every polling cycle, the Solver also checks for issues labeled `in-progress` where the start timestamp (from the comment) exceeds `--timeout`. If stale:
- Replace label `in-progress` → `failed`
- Comment explaining the timeout
- Clean up worktree if it exists

The human decides what to do with `failed` issues — re-approve, refine further, or close.

## Configuration

Via environment variables or `.env` file:

```bash
REPO="jpfaria/OpenRig"           # Target repository
REFINER_INTERVAL=300              # Refiner polling interval (seconds)
SOLVER_INTERVAL=600               # Solver polling interval (seconds)
SOLVER_TIMEOUT=3600               # Max time for in-progress before failed (seconds)
CLAUDE_MODEL="claude-sonnet-4-6"  # Claude model for analysis
```

## Dependencies

- `gh` — GitHub CLI (authenticated)
- `claude` — Claude Code CLI
- `git` — Git
- `jq` — JSON processing

## Future: Kubernetes Deployment

Both scripts are designed to run as standalone containers:
- One pod for refiner, one for solver
- Configuration via ConfigMap/env vars
- `gh` auth via mounted secret
- `claude` auth via mounted API key
- Persistent volume for worktrees (solver)

## Limitations

- One issue at a time per solver instance (sequential processing)
- Requires `claude` CLI available in PATH
- Requires `gh` CLI authenticated with repo access
- Labels must exist in the target repository
